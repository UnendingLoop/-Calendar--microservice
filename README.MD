# Calendar Service

Учебный HTTP-сервис для управления событиями пользователей в формате календаря.  
Поддерживает CRUD-операции + выборку событий за определенный период.  
При запуске приложения данные считываются из файла и хранятся в оперативной 
памяти в конкурентном доступе, при завершении работы сохраняются в файл.

## Возможности

- Создание/Чтение/Обновление/Удаление события
- Привязка события к конкретному id пользователя(без системы авторизации)
- Получение списка событий:
  - на день;
  - на неделю;
  - на месяц;
- Валидация входных данных на уровне хендлера и сервиса;
- Чтение активных и архивных событий из файла (JSON) при запуске и сохранение 
 при завершении работы приложения(имена файлов можно задать в .env);
- 2х-уровневое (request start/end info, error) логирование всех HTTP-запросов (middleware) 
через канал с выбором вывода (файл/stdOut, задается в .env); 
- Периодическая очистка прошедших событий (интервал очистки задается в .env)


## Архитектура

HTTP-часть сервиса разделена на 4 слоя:
- **logger** - mw для логирования каждого запроса
- **transport** - парсинг запросов, формирование HTTP-ответов
- **service** - бизнес-логика, проверки входных данных, ошибки
- **repository** - работа с in-memory картами событий:
    - активные в **eventMap map[uint][]model.HeapEntity**, 
    - архивные в **archive  map[uint][]model.Event**,
и кучей, реализующей сортировку событий по времени их 
наступления (методы кучи описаны в **model**).

Так же присутствуют следующие модули:
- **cleaner** - запускает горутину, которая периодически чистит "выполненные" события из карты активных событий.
- **engine** - фабрика экземпляра сервера(*http.Server), определяет API-эндпоинты.
- **model** - определяет модели данных, используемых внутри приложения.
- **notifier** - запускает горутину, которая:
    - при запуске приложения получает ближайшее событие и засыпает до времени его наступления;
    - параллельно слушает канал синхронизации, и при поступлении сигнала пересчитывает таймер сна до ближайшего событие;
    - при наступлении времени ближайшего события помечает его "выполненным", и снова пересчитывает таймер сна.
- **storage** - реализует считывание и запись событий при старте/завершении приложения.

Реализован Graceful Shutdown по **os.Interrupt** и **syscall.SIGTERM**.

## Модели данных

### Event
```go
type Event struct {
	EID         string      `json:"event_id"`                   // uuid задачи/события
	UID         uint        `json:"user_id" binding:"required"` // id создателя события
	Created     time.Time   `json:"created"`                    // дата создания задачи/события в UTC
	Updated     time.Time   `json:"updated"`                    // дата обновления события/задачи пользователем в UTC
	Scheduled   *CustomTime `json:"date" binding:"required"`    // дата выполнения/наступления задачи/события
	Description string      `json:"event" binding:"required"`   // сам текст события/задачи
	IsDone      bool        `json:"is_done"`                    // флаг "выполненности" задачи/события
}
```
Во входящих запросах время для создания/обновления событий ожидается в формате "YYYY-MM-DD HH:MM"(model.CustomTime),
в то время как для запроса списка событий за день/неделю/месяц ожидается формат "YYYY-MM-DD". 
**NB:** 
- Указанному в запросе времени назначается часовой пояс сервера. 
- Внутри приложения всё время конвертируется в UTC.
- Валидное время в запросе в поле "Scheduled" при создании/обновлении события - только в будущем.

### Heap
```go
type HeapEntity struct {
	Index int // для синхронизации события из карт с его индексом в куче
	Event *Event
}

type EventHeap []*HeapEntity
```

## Запуск

Предварительно создать .env из файла-примера в корне проекта, внести в него необходимые правки:
- APP_PORT - порт для запуска сервера приложения.
- FILENAME - название файла для чтения и записи активных событий при старте/завершении работы.
- ARCHIVE - название файла для чтения и сохранения архивных событий при старте/завершении работы.
- GIN_MODE - режим запуска сервера("release"/"debug").
- LOG_MODE - режим логирования внутри приложения:
    - "" - при пустом значении используется stdOut для вывода;
    - "stdout" - логгер будет писать в stdOut;
    - "someFileName" - логи будут записаны в файл, он будет создан в рабочей директории 
    с названием "[текущая дата]someFileName".
- CLEANER_FREQ - задает периодичность запуска очистки выполненных событий.


Перейти в корень проекта и запустить:
```bash
go run ./cmd/main.go
```

## API list

1. Создать событие - POST /events
Формат JSON:
```json
{
  "user_id": 1,
  "date": "2026-12-31 16:00",
  "event": "Новый год"
}
```
Ответ 201 Created:
```json
{
    "event_id": "12fd5d91-8511-481c-8894-0900943b27d3",
    "user_id": 1,
    "created": "2026-03-23T19:17:58.308922137Z",
    "date": "2026-12-31 16:00",
    "event": "Новый год",
    "is_done": false
}
```

2. Обновить событие - PATCH /events
Формат JSON:
```json

    "user_id": 1,
    "date": "2027-01-14 00:00",
    "event": "Старый новый год"
}
```
Ответ 200 OK:
```json

    "event_id": "12fd5d91-8511-481c-8894-0900943b27d3",
```

## Статус-коды

Сервис использует следующие коды ответов:
- 200 OK/201 Created/204 No content — успешные запросы
- 400 — ошибки ввода (некорректная дата, пустые поля и т. д.)
- 404 - если событие/пользователь не найдены
- 500 — системные ошибки

## Примеры cURL-запросов

- создание события:
```bash
curl -X POST http://localhost:8080/events \
  -H "Content-Type: application/json" \
  -d '{
    "user_id": 1,
    "date": "2026-07-01",
    "event": "some event"
  }'
```

- обновление события:
```bash
curl -X PATCH http://localhost:8080/events \
  -H "Content-Type: application/json" \
  -d '{
    "event_id": "PUT-YOUR-UUID-HERE",
    "user_id": 1,
    "date": "2025-01-02",
    "event": "Обновлённая задача"
  }'
```

- удаление события:
```bash
curl -X DELETE http://localhost:8080/events \
  -H "Content-Type: application/json" \
  -d '{
    "event_id": "PUT-YOUR-UUID-HERE",
    "user_id": 1
  }'
```

- получить события на день:
```bash
curl -X GET "http://localhost:8080/events_for_day?user_id=1&date=2025-01-01"
```

- получить события на неделю:
```bash
curl -X GET "http://localhost:8080/events_for_week?user_id=1&date=2025-01-01"
```

- получить события на месяц:
